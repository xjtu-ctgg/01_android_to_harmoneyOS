以下是来自各参赛选手的问题收集：

l 问：可以跨地域 和 跨部门组队吗？
答：可以，但是部门和地域统计是按照队长的部门和地域统计。

l 问：报名时间可以到7月10号？
答：是的，预赛答题阶段也可以报名，直到预赛提交截止之前。

l 问：必须是队长来报名吗？
答：是的，平台要求队长来报名，默认是报名人员为队长，且不能更换队长，可以修改成员信息。

l 问：OD、外协员工、实习生、引望无华为工号员工可以参赛吗？

答：可以，但是必须和华为自有员工组队报名（主要是因为涉及将来获奖后奖金的打款问题）。

● 会不会出现某个队在两个赛道都进了决赛？

回答：一个队伍可以同时参加2个赛道，并且分别获奖，我们期待出现一个队伍获得“双王”的出现；

详细内容请参考比赛指导

● 预算和决赛的题目都是一样的吗？

回答：难题赛道不设决赛，入围TOP18的作品会通过在线答辩的方式给出主观打分，结合客观打分决出名词；

赛题赛道设置决赛，题目与预赛不一样，会提前公布赛题的方向。具体赛题的详细情况在决赛现场给出；

详细内容请参考比赛指导

● 那进入决赛的最多能有几个队伍

回答：每个赛道都有36名获奖选手，同时每个赛道有36个队伍突破预赛进入下一轮。也就是进入下一轮即已获奖；

详细内容请参考比赛指导

● 是5个题目都要做？还是5选1？

回答：一个队伍在一个赛道内，选择任意一个题目参赛即可。当然也不限制参与多个题目，在这种情况下，我们会以最优秀的作品作为该队伍的参赛结果与其他队伍进行PK。

详细内容请参考比赛指导

● 最后的评价标准是怎样的呢？验收用例通过多少？这个到时候是有个平台可以自动去报跑和验证吗？

回答：

1、评分分为两部分，客观分与主观分，客观分是以通过实际通过用例的数量来决定的。这里包括：准确性得分 + 稳定性得分；

准确性得分：5次运行，以最多通过的用例数为结果；

稳定性得分：5次运行，以5次都通过的用例数为结果；

极端情况下100个用例，可能出现准确性为满分，稳定性为零分；

 

入围赛在作品通过人工审核后，以客观分为基础；详细内容请参考比赛指导

 

● 作品要求和题目内的交付件是要一起提交吗？

回答：是的

回答：work+INSTRUCTION.md用于平台运行作品，做客观分评估；/log、/result目录用于存放选手自己提交的执行记录，作为评审组主观评判的依据 。

回答：

l 本地如何调试来模拟提交后的运行环境以保障提交后能够按照预期运行？

回答：大家可以在本地将INSTRUCTION.md作为Prompt的内容去调试work。

 

l 测试平台运行时长有限制么？

回答：会有的，具体的时长判题系统上线时会公示，超长的任务在性能项上得分上会比较吃亏。

 

l 请问什么时候开放跑分平台？

回答：6月22日

 

l 平台跑分这个客观分评分有标准吗，只看用例通过率还是推理耗时、token数都会看？

回答：客观分主要看用例通过的情况，有准确性和稳定性，准确性看通过率最高的一次，稳定性看5次均通过的用例的比率。耗时/Token消耗会作为主观评分依据。

 

l 交付件这个目录结构下，平台是怎么扫描并加载Skill/Agent/MCP的呢？

回答：平台会读INSTRUCTION.md作为入口，你需要在里面指明Agent如何使用你的WORK，其他的result和log主要是给评委主观评分使用的。

 

l 主观评分 和 平台客观跑分，两者占最终得分的比例是怎样呢？

回答：客观分60，准确性30，稳定性30，主观分数40，其中10分是性能（根据你的作品在同一题中的性能排序参考）

 

l 平台上运行的codeagent是什么版本的？

回答：codeAgent2.0

大赛答疑会相关问题摘录：                                                                                                                          

l Q1: 报名只能报一个赛道吗？                                                                                          

  ▎ 可以改报两个赛道。报名信息可以修改，目前的报名主要是为了数据统计和定向邀请。

 

l Q2: 可以同时做两个赛道的题目吗？

  ▎ 可以。每个赛道的题目都可以做，极端情况下可以全部都做，最终以提交的作品为准。

 

l Q3: 每个赛道有多少入围名额？

  ▎ 每个赛道前36名入围，入围即获奖。奖项设置为：一等奖3名、二等奖6名、三等奖9名、优胜奖18名。

 

l Q4: 自由命题（创意赛道）和其他赛道的评分标准有什么不同？

  ▎ 客观分数上限不同，其他题目客观分按60%，创意赛道有"创意价值"评分项。决赛难题赛道入围后的前18名需要在线答辩展示。

 

l Q5: 题目有疑问在哪里提问？

  ▎ 可以通过isales提issue的方式，或在答疑群里反馈。题目目前处于试运行阶段，会根据大家的问题做调整并公示。

 

l Q6: 题目后面会有调整吗？

  ▎ 是的，部分题目发布后近期会根据大家疑问做细微调整，以最新下载版本为准。

 

l Q7: 验收标准是什么？如何自验证？

  ▎ 验收标准是修复报告。自验证方法：将instruction.md在本地运行，看能否输出符合题目要求的结果。

 

l Q8: 可以使用embedding模型吗？

  ▎ 暂时部署不了。如果是CPU上能跑的embedding模型，可以放在work里指定加载方式，但需要提供测试平台可访问的地址。也可以用开

  源模型部署接口。

 

l Q9: 使用什么模型？deepseek pro还是flash？

  ▎ 目前只能提供flash，pro版本因算力原因暂不提供。

 

l Q10: 可以控制模型的温度、思考力度等参数吗？

  ▎ 控制不了，使用的是目标系统自带的那一档配置。

 

l Q11: 评分标准是什么？

  ▎ 准确率（5次中得分最高的）+ 稳定性（所有检查点都过的）。如果稳定性和准确率都高，分数就优秀。

 

l Q12: 运行时长和token消耗占多少比例？

  ▎ 占10%，作为参考给评委，同一题目内做横向PK。不作为客观评分。

 

l Q13: 提交后怎么判题？

  ▎ 平台会自动运行5次，不是手动提交5次。提交后可以看到客观分数、运行时长、token消耗。

 

l Q14: 每天可以提交几次？

  ▎ 手工提交每天每道题最多5次。平台内部会跑5个实例。

 

l Q15: 如果遇到环境问题导致执行失败怎么办？

  ▎ 如果是工具问题不计入；如果是自身问题会计入。遇到问题可以反馈给赛题组协助解决。

 

l Q16: 验证环境是什么系统？Python版本？

  ▎ Linux环境。Python等基础工具会有，具体版本平台上线时会公示。验证环境是一个黄区docker，无外网权限。

 

l Q17: 需要用数据库怎么办？

  ▎ 建议用本地能部署的数据库（如sqlite）。如需外部数据库，自己搭建并确保网络可达。

 

l Q18: 第三方包如何处理？

  ▎ 建议把第三方包放到work目录，并在instruction.md第一阶段指定安装。

 

l Q19: 重构题目需要提供什么？

  ▎ 需要提供重构的prompt。验证点有两个：重构后的结果 + 重构的过程。

 

l Q20: 重构题目太复杂跑不完怎么办？

  ▎ 题目已做简化：可能缩小代码量（如从10万行缩到1万行），或提供部分框架，只完成指定功能即可。

 

l Q21: 决赛怎么比？

  ▎ - 竞赛赛道：36名选手8月1日当天到研究所现场比赛

  ▎ - 难题/创意赛道：36名中选18名进行在线答辩（10-20分钟）

 

l Q22: 跨地域组队可以吗？

  ▎ 不建议。比赛时需在同一研究所同一台机器上操作。

 

l Q23: 决赛环境需要自己搭建吗？

  ▎ 不需要，机器和算力由大赛提供。

 

l Q24: 可以使用开源工具/方案吗？

  ▎ 可以。外部有成熟经验或方法可以直接使用。

 

l Q25: 可以用claudcode吗？

  ▎ 不可以，公司内部对claudcode有限制。

 

l Q26: 自主命题可以用Windows平台吗？

  ▎ 可以。如果作品是Windows平台运行的，说明清楚，届时由对应团队人工评估

 

l 如果跨研究所，最后线下集中，是否可以各自在各自的研究所，然后拉会交流一起搞？还是必须去同一个研究所下线一起搞呢

回答：竞赛赛道决赛选手各自在所在的各研究所/东莞/深圳线下集中比赛，同一个战队不能跨地域参赛。

 

l 请问赛题有蓝区的版本吗？

回答：当前赛题来自真实业务问题，暂时不放到蓝区，建议在绿区或黄区参赛。

 

l 请问 INSTRUCTION.md 是否有参考基线？根目录的md描述中提到“平台通过加载该Markdown执行作品运行过程，验证用例通过性”，对于c_to_rust 是否需要包含 执行改造，编译构建，测试用例执行引导，性能测试及对比（引导构建c，rust两个平台的性能测试并对比），Rust unsafe 比例统计 这五部分的prompt吗？判题时只执行构建，功能/性能测试，和Rust unsafe 比例统计这三部分吗？

回答：会有构建的部分，因为要跑DT的用例，题目要求里面有重构后要通过原有的test目录下的用例验证，这部分用例也是需要重构的，确保防护网拦截一致
