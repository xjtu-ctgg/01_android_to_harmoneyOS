# 隐藏用例对抗测试与二次优化

更新日期：2026-07-13

## 1. 本轮结论

现有仓内测试已扩展到 197 项（最终数字以 `tools/verify.sh --static` 的最新输出为准），并形成 178 条 Journey（22 条核心映射、156 条状态边界与视觉变体）。新增门禁不是对“满分”的替代证明，而是针对平台公开的功能一致性、截图相似性、接口一致性和代码规范四个维度补强可重复证据。

本轮发现两个值得修复的真实差异：

1. Android `FilterScreen` 的选项状态属于组件本地 `remember` 状态，关闭后重新创建会回到默认值；鸿蒙实现此前只重置排序和卡路里，会保留价格、分类及生活方式选择。现已在 `showFilters()` 中同步清空选择集。
2. 暗色主题下 ArkUI 资源会切换，但系统状态栏和导航栏前景色此前固定为黑色。现已根据 `Configuration.colorMode` 同步使用浅色或深色系统栏前景，并在运行时配置变化时更新。
3. Detail Hero 此前按固定 360vp 视口放置，窄屏时会偏离 Android 的 24vp 水平边距；现改为读取实际组件宽度，动态计算展开图片尺寸及展开/折叠 X 坐标。
4. Cart 侧滑背景此前始终占满 132vp 高度；现按 Android `SwipeDismissItemBackground` 的进度区间生成方形 Surface、圆角、4vp 内边距、图标透明度和文字切换。
5. Feed/Cart 的滚动内容此前在 64vp 后开始，而 Android 明确使用状态栏加 56vp 的内容占位并让 DestinationBar 固定覆盖；现已恢复 56vp 起点和固定覆盖层结构。
6. 高亮商品卡的六卡渐变此前只按卡片索引定位，水平滚动后不会移动；现按 Android `left - scroll/3` 公式加入滚动视差。
7. Search 分类网格此前每行固定 128vp，平板会裁切大卡；现按实际视口、两列边距和 1.45 宽高比计算网格高度。分类图也从固定 134vp 改为至少 134vp、随卡高增长，并把分类/建议行改为最小高度以适配大字体。
8. 根布局此前固定上下各 24vp，无法覆盖刘海、三键导航、手势指示条和自由窗体。现读取 `TYPE_SYSTEM` 与 `TYPE_NAVIGATION_INDICATOR`，监听 `avoidAreaChange`，把实时 px 转为 vp 后用于根布局，并在 Ability 销毁时注销监听。实现方式与 HarmonyOS 官方 ResponsiveLayout 示例一致。
9. Cart 错误 Snackbar 此前需要点击才会消失，与 Android `SnackbarHostState.showSnackbar()` 默认短时提示不一致；现按 AndroidX Compose 源码的 `SnackbarDuration.Short = 4000ms` 自动消失，并用 generation 防止旧定时器清除后显示的新消息。
10. 字面 `.id()` 唯一并不能证明 `ForEach` 展开后唯一；现根据 5 个 Feed 集合和重复商品数据展开卡片、图片、集合与动作 ID 做冲突检查，并增加从第 3 集合的 `snack.card.3.1` 进入 Cupcake 详情的接口 Journey。
11. 仅使用 `TYPE_SYSTEM` 的顶部/底部矩形仍会遗漏横屏侧边刘海。华为官方 ResponsiveLayout 示例把 `TYPE_CUTOUT` 独立读取并监听；现同步合并 system/cutout/navigation 的四边最大避让值，并覆盖 844×390 横屏左侧 44vp 挖孔场景。
12. Cart 错误消息资源此前只是存在但未被运行时代码引用，仍可能命中硬编码用户可见字符串规则。现用 `Resource` 保存消息、用独立可见状态控制生命周期，两条失败路径均直接引用 `cart_increase_error`/`cart_decrease_error` 资源，且避免 `ResourceStr` 在 ArkTS `Text` 重载上的联合类型歧义。
13. Feed 集合箭头此前通过模板字符串硬编码英文无障碍说明。现新增 `view_collection = "View %s"` 格式化资源并由 `$r(...)` 引用，同时增加仓级门禁防止 Cart 错误或集合说明重新退化为硬编码用户可见文案。
14. Search 无结果和 Profile 的空状态 SVG 原始尺寸为 341×179；ArkUI 此前固定 341vp，在 320vp 屏幕扣除左右 24vp 后会超过可用宽度。现使用 `width('100%') + maxWidth:341 + aspectRatio(341/179)`，增加两条 320×720 截图 Journey。
15. Cart 标题此前把所有数量写成 `Order (%d items)`，与 Android plurals 在 1 行时的 `1 item` 不一致。现使用 HarmonyOS 原生 `plural.json` one/other 资源，并覆盖删除两行后只剩一行的文本、Subtotal 和 Total。
16. `core.yaml` 此前仅能被正则读取，flow list 内未加引号的冒号 token 会被标准 YAML 解析器拒绝；现统一引用并用 Ruby/Psych 验证 59 条均可解析。同时修正不存在商品 `Zucchini` 的错误正向断言为真实 NoResults 状态。
17. Cart CheckoutBar 此前比 Android 源码高：鸿蒙按钮为 48vp、列表底部余量为 88vp；Android 的 `JetsnackButton` 由 Material3 默认值形成 40dp 按钮，外部上下各 8dp，总高 56dp，列表末尾也明确为 56dp。现已逐项恢复并增加固定视口截图 Journey。
18. Cart 标题此前误用 28vp 行高，Summary 还用统一 40vp 行高和单条 Divider，导致截图纵向节奏与 Total 对齐方式偏离源码。现恢复 titleLarge 24vp 行高、56/28/44/44vp 几何、Total 右对齐以及上下两条 Divider。
19. FilterOverlay 此前 chip 使用 16/7vp 内边距，并对各区段普遍追加 20vp 顶距；Max Calories 甚至误用 12sp。现按固定节点源码恢复 20/6vp、section 间距和 20sp/24sp titleLarge。二次核查 `Shape.kt` 后确认 Jetsnack 自定义 medium=20dp、small=50%，因此 panel/chip 最终保持 20vp/胶囊形，而不是 Material3 默认 Shapes。
20. Cart 侧滑此前以固定 180vp 计算 progress，并用 120vp 位移阈值删除；Material3 源码实际按组件全宽计算 progress，默认位移阈值 56dp、速度阈值 125dp/s。现按动态视口宽度与双阈值实现，并覆盖阈值前后和快速 fling。
21. Android 的 Cart 使用逻辑方向 `EndToStart`，在 RTL 下应由左向右滑；鸿蒙此前固定 `PanDirection.Left`。现用 LocalizationKit 运行时判断系统语言方向，分别接受 LTR 左滑与 RTL 右滑，并覆盖 RTL 正向、阈值及反方向忽略。
22. Search group header 和 suggestion row 旧实现用 16/12vp 垂直 padding 模拟正常字体居中，但在 2.0 字体下分别膨胀到约 80/72vp；源码是 minHeight 56/48vp、header 仅 4vp 垂直 padding、row 无垂直 padding。现改用 Row 垂直居中并恢复每组末尾 4vp Spacer。
23. Search Add、Cart Checkout、Detail Add to Cart 旧实现均使用纯品牌色，而 Android `JetsnackButton` 默认使用随主题切换的 `interactivePrimary` 水平渐变；Detail 还误写为 48vp 按钮。现新增 base/dark 渐变资源，三处统一恢复渐变，Detail 恢复 40vp 胶囊。
24. Quick Filter、FilterOverlay Chip 与筛选图标旧实现使用单色 `brandSecondary` 边框；Android 实际使用随主题变化的 `interactiveSecondary` 对角渐变。现用 `borderImage` 恢复 base/dark 渐变，并冻结 24vp 图标边框与 48vp hit target 的嵌套关系及选中态无边框语义。
25. 根布局此前统一把 bottomInset 放在 BottomNav 外侧，导致手势导航区显示页面背景；Android 的 `navigationBarsPadding()` 位于品牌色 BottomBar Surface 内。现由 BottomNav 自身吸收底部 inset，Detail 无 BottomNav 时才由根布局避让，并分别覆盖两种安全区颜色。
26. Android QuantitySelector 的两个按钮不是 48dp 可见圆：`JetsnackGradientTintedIconButton` 直接包裹默认 24dp Icon/Surface，以 2dp `interactiveSecondary` 渐变描边和渐变图标显示，触摸系统再扩展命中区。鸿蒙此前把 48vp 命中尺寸误当成可见/布局尺寸，并用单色边框。现以 24vp 可见节点配合 48vp `responseRegion`，按明暗主题恢复渐变；为保持接口一致性，新增专用 quantity SVG 而不改变共享中性 add/remove 资源。Cart 与 Detail 各增加一条固定主题截图 Journey。
27. Profile 已有几何、排版和明暗 VectorDrawable→SVG 逐路径合同，但 Journey 此前只有默认/320×720 窄屏，未覆盖 dark 与低高度大字体的组合。新增 360×800 dark 和 320×480、2.0 字体两条截图 Journey，验证主题资源选择及居中内容在紧凑高度下不重叠；源码无差异，因此不修改生产布局。
28. DestinationBar 此前把 Compose `TopAppBar` 误近似为 63vp 内容+1vp Divider，并使用 24/16vp 横向 padding 与不透明背景。比赛节点的 Material3 runtime token/布局常量表明小型容器为 64dp、框架水平 padding 4dp、无导航图标时 title inset 12dp；源码还明确使用 `AlphaNearOpaque=0.95`。现恢复 64+1vp、16/4vp 和明暗 95% overlay，并增加滚动后 Feed/Cart 两条截图 Journey，覆盖内容从顶栏下方经过时的混色差异。
29. BottomNav 的终态几何已有结构合同，但原 RTL 数量选择器 Journey 位于 Detail（底栏隐藏），不能覆盖 RTL 底栏布局；暗色也只有页面级截图，没有冻结选中/未选中 tint。新增 dark Cart 与 RTL Feed 两条截图 Journey，验证暗色 brand/interactive/inactive 色值、56vp 内容高度、逻辑顺序镜像及指示器跟随；生产布局对照无差异，故不引入可能降低截图确定性的额外动画实现。
30. 普通 SnackItem 被固定为 136×168vp并强制标题单行省略，但 Android 横向 LazyRow 给子项横向非约束测量，120dp 图片仅形成最小宽度，`wrapContentWidth()` 会让长标题扩展卡片。现恢复内容宽度并将普通 List 改为最小 176vp而非固定高度，兼顾默认尺寸和 2.0 字体增长；同时移除 Highlight tagline 源码不存在的单行限制。新增 Detail Related、Cart Inspired 滚动到 “Ice Cream Sandwich” 的两条截图 Journey，断言完整标题和无 ellipsis。
31. Search Result 的 Android Row 没有对 name/tagline 设置 `maxLines`，依赖内容测量适应窄屏与字体缩放；鸿蒙原实现把两者都限制为一行。现恢复自然换行，并增加精确查询 “Ice Cream Sandwich” 与 320×720、2.0 字体 Apple 结果两条 Journey，验证完整标题、结果行增高、按钮仍可见且不重叠。SearchBar 本体经源码核对无稳定终态差异，保持不变。
32. Detail Title 源码以 `heightIn(min=128dp)` 承载可增长的 headline/tagline/price，SEE MORE/LESS 则是 `heightIn(min=20dp)` 配合 15dp 顶部 padding；鸿蒙此前分别固定为 128vp 与 44vp。现恢复两处最小高度语义，新增 320×720、2.0 字体长标题详情截图和展开控件默认约 31vp 高度截图，避免标题裁切及 Ingredients 垂直偏移。
33. FilterOverlay 原固定450vp，而 Android 是屏幕外边距16dp与 `heightIn(max=450dp)`；短视口下固定值会越过 BottomNav 上方可用区域。现使用内容区高度减32vp并封顶450vp，标准终态不变、紧凑屏保持上下16vp；增加360×400精确288vp面板和2.0字体滚动到Lifestyle两条Journey，覆盖容器截图与末端控件可达性。
34. CartItem 原固定132vp并截断商品名，图片在增高场景下垂直居中；Android则由100dp图片的16dp上下padding形成自然最小高度，name可换行，tagline使用28dp bodyLarge。现恢复最小高度、顶部对齐、图片top/bottom inset、标题换行和tagline指标；用初始订单中的“Ice Cream Sandwich”增加默认与320×720、2.0字体两条截图Journey。
35. Cart标题和Summary默认几何虽已对齐，但仍把Compose的`heightIn`误写为固定高度。现将订单/Summary标题改为最小56vp的垂直居中Row，三种金额行改为最小自然行高加padding；新增2.0字体订单标题与Summary两条Journey，验证标题可见、Subtotal/Shipping/Total增高且无重叠。
36. Cart Checkout和Detail Add to cart把JetsnackButton的40dp默认最小高度与底栏56dp最小高度误写为固定值。现恢复两层最小高度，并保持按钮文字单行；新增两条2.0字体底栏Journey，验证按钮文字、Quantity标签和底栏增长均无裁切/重叠。
37. Feed集合标题与FilterBar把源码的56dp最小高度误写为固定高度。现恢复`minHeight:56vp`，新增2.0字体集合标题及3.0字体FilterBar两条Journey，覆盖长文本/极大字体下的增长、筛选按钮可见和无重叠。
38. Filter Header与SortOption分别把Compose IntrinsicSize/自然测量误写为固定48/38vp。现把Header父子、Close/Reset触控节点和排序行恢复为最小尺寸；新增2.0字体Header与SortOption两条Journey，验证标题、Reset、图标、check和长排序文字无裁切。
39. SearchCategory源码不限制文字行数，而是在55%固定宽度中真实测量、垂直居中并由卡片clip；鸿蒙原`.maxLines(2)`改变了大字体可见区域。现移除限制，增加2.0字体类别文字与RTL文字/图片逻辑位置两条Journey。
40. Cart Snackbar 的宿主时长虽然已对齐，但内容 Surface 仍与固定 Android/Material3 源码不同：旧实现为全宽红底、白色 Karla Bold 且点击整面会关闭；现恢复12dp外边距、68dp双行最小高、16/8/6dp内边距、50%胶囊、`uiBackground`、`textSecondary` 与 Montserrat Medium，并增加明暗截图及“触碰不关闭”Journey。
41. （已纠正）早期只检查 `Search.kt` 调用点，漏读 `model/Search.kt`，因而把 Search 错判成同步；固定提交中的 `SearchRepo.search()` 实际为 suspend 函数并执行 `delay(200L)`。当前已恢复200ms LoadingProgress窗口与 generation 取消语义，所有结果 Journey 等待250ms，并覆盖100ms加载态、连续查询、切Tab和清空时旧查询不得回写。
42. Snackbar 视觉节点虽已对齐，但旧鸿蒙结构把它放在 `checkoutBar()` 的 Column 中，会把 Checkout 向上推；Android 的 `JetsnackScaffold → Material3 Scaffold` 则把 Snackbar 作为内容覆盖 sibling，直接覆盖内容区底部的 Checkout。现移到 Cart 根 Stack 最后绘制并更新固定截图断言，防止“形状正确但整块垂直位置错误”。
43. Snackbar 连续消息和宿主生命周期此前仍不一致：Android `SnackbarManager` 追加FIFO，四个主Tab共享根Host；进入Detail销毁当前Host，但未展示队列保留至返回。鸿蒙现用Resource队列逐条显示，把提示节点提升至Index共享Stack，并在Detail往返时丢弃当前显示、保留和恢复下一条；增加五条竞态/跨页面Journey。
44. SearchState此前被当成可跨页面/旋转持久状态，但固定源码只使用`remember`而非`rememberSaveable`。现区分生命周期：同一Tab重复选择保持；离开Search、进入Detail或配置重建后重置为空query、未聚焦Categories。对应详情返回和旋转Journey由“保持Apple”更正为“恢复初态”，新增Cart往返重置Journey，并移除所有300ms等待。
45. Filter状态此前被统一当成Overlay本地状态，重开时错误清空所有Chip；源码实际把Quick、Price、Category、Lifestyle四组Filter对象放在进程级全局列表，仅sort与maxCalories使用Overlay内plain `remember`。现Overlay重开只重置sort/slider并保留Chip，Quick Filter独立提升为进程期状态；新增跨Tab、Detail和旋转5条Journey。
46. Quick Filter旧实现固定14vp圆角，只在默认28vp高度时等价于胶囊；Android `Filters.kt + Shape.kt` 使用随控件高度变化的small=50%。现改为50%动态圆角，并用3.0字体light未选与dark选中截图冻结高度增长后的胶囊终态。
47. Search Result Add视觉为36dp，但Compose会自动把小于48dp的clickable触控目标向外扩展；鸿蒙旧实现命中区只有36vp，边缘点击可能冒泡打开Detail。现用48vp responseRegion恢复触控合同，并新增视觉圆外5vp点击保持Search、Cart不变的接口Journey。
48. Search Clear位于40dp内容行，Android IconButton仍有48dp高触控目标；鸿蒙旧实现仅48×40vp。现向上下各扩4vp，并增加顶部视觉边界外3vp点击后query清空、焦点保持、Suggestions出现的接口Journey。
49. Quick/Overlay Filter Chip默认约28dp可见高但Compose toggleable有48dp最小触控目标。鸿蒙现按运行时实测高度计算`max(48,height)`与居中偏移，默认字体扩展、大字体不缩小；增加上下视觉边缘外8vp点击用例，并验证不误触父列表或scrim。
50. SortOption默认约38dp可见高但Compose selectable触控目标至少48dp。现按三项索引分别跟踪运行时高度并独立扩展，避免大字体长默认文案和短Rating互相覆盖；新增Rating底缘外点击及Reset启用状态Journey。
51. Detail Back是36dp可见圆、SEE MORE/LESS默认约31dp高，但Compose触控目标均至少48dp。现Back固定居中扩展，展开控件按实测高度动态扩展；新增左侧/底部视觉边缘外点击用例，分别验证返回Feed与展开详情。
52. Cart CHECKOUT和Detail ADD TO CART可见最小高40dp，Compose交互仍至少48dp。现按各自实测高度动态扩展，新增按钮上下可见边界外3vp点击，并确认两项源码TODO/no-op不改变Cart或Detail状态。
53. Detail 数量和 SEE MORE/LESS 都是 Android 普通 `remember`，配置重建后应恢复数量1与折叠态；新增先修改两项状态再旋转的 Journey，防止鸿蒙全局 Store 错误保留瞬态界面。
54. Detail 最底部源码余量为56dp底栏加8dp内容空隙、导航安全区另算；将错误的80vp改为64vp，并新增滚到最后一个相关推荐集合的安全区截图 Journey。
55. Cart Checkout 的 Android 源码是物理 Left 对齐，不能直接映射成会随 RTL 翻转的 ArkUI Start；仅对该文本固定 LTR 后使用Start，新增 RTL 按钮半区与文字物理左侧截图。
56. Detail 折叠 Hero 的 Compose `placeRelative` 表示逻辑End，RTL 下应位于左侧24vp；现按系统 RTL 信号镜像折叠终点，并新增295vp滚动后的150vp图片定位截图。
57. 增加 Cart 数量跨Tab、旋转保持，以及带query/fragment的暖启动深链三组黑盒 Journey，覆盖 ViewModel 等价生命周期和入口后缀解析。
58. 增加 `force_stop + cold_launch` 进程重建矩阵，验证 Cart 数量、Quick Filter 与 Overlay Chip 均恢复冷启动初值，不把进程内状态错误写成持久化状态。
59. 图片资源虽然已有逐字节 SHA 门禁，但此前没有用极端纵横比验证列表/购物车进入详情后的裁切稳定性。现增加1716×2288 Gingerbread与2100×1182 Froyo两条跨页面Journey，分别冻结Cart 100vp、Feed 120vp与Detail 300vp圆形容器中的Cover中心裁切和无拉伸；结构合同同时锁定Android `ContentScale.Crop`与Harmony `ImageFit.Cover`映射。

## 2. 官方资料核验及其对测试设计的影响

- HarmonyOS/OpenHarmony UiTest 的 `ON.id()` 可以按组件 ID 建立选择器，`findComponent()` 和 `waitForComponent()` 可定位节点。因此稳定且唯一的 `.id(...)` 是接口用例可执行性的直接基础。[OpenHarmony UiTest API](https://gitee.com/openharmony/docs/blob/master/zh-cn/application-dev/reference/apis-test-kit/js-apis-uitest.md)
- UiTest `Component.getBounds()` 可读取控件边界，适合检查详情 Hero 折叠后的 150vp 图片、56vp 标题位置、触控区域及遮挡关系；`isClickable()` 可验证动作节点是否真实可点击，而不只是存在文本。[OpenHarmony UiTest API](https://gitee.com/openharmony/docs/blob/master/zh-cn/application-dev/reference/apis-test-kit/js-apis-uitest.md)
- UiTest `screenCap()` 可把指定屏幕保存为 PNG；因此 Journey 明确固定冷启动、主题、字体比例、等待时长和截图点，减少截图比较中的非业务噪声。[OpenHarmony UiTest API](https://gitee.com/openharmony/docs/blob/master/zh-cn/application-dev/reference/apis-test-kit/js-apis-uitest.md)
- ArkTS 通过更严格的静态类型与语法约束把一部分错误前移到编译期，所以公共 API 编译仍是必要门禁，但不能替代官方 Code Linter 的规则扫描。[华为 ArkTS 概览](https://developer.huawei.com/consumer/en/arkts/)
- 华为官方文档将 Test Kit 定位为同时提供单元测试和 UI 测试能力；本地合同测试负责确定性事实，设备 UiTest 负责运行时节点、边界、点击与截图，两者不能互相冒充。[HarmonyOS 文档中心](https://developer.huawei.com/consumer/cn/doc/)

## 4. 仍需评分环境提供的最终证据

以下项目无法仅凭本机静态测试或公共 OpenHarmony SDK 宣称通过：

1. 正式 HarmonyOS SDK 下的完整构建和签名安装。
2. 官方 Code Linter 全量规则报告为非空且无评分规则对应告警。
3. 真机或模拟器上按稳定 ID 执行 Journey，并取得实际 PNG。
4. 将同尺寸、同主题、同字体比例的 Android 与 HarmonyOS 截图做像素或感知差异比较。

因此当前策略是尽可能提高命中率，但不承诺不可见样例必然满分。平台环境应先运行 `tools/verify.sh --strict`，再执行 Journey 的节点、边界和截图断言。
