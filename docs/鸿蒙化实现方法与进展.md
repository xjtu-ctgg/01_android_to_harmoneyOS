# Jetsnack 安卓 APP 鸿蒙化：实现方法与进展

> 建立日期：2026-07-11  
> 当前状态：主应用与 Recent Orders Form Kit 迁移、Locale/货币一致性、API 20 公共兼容构建、五 Executor 本地复现及最终 ZIP 全新解压复验已完成；现代专有 Code Linter、真机截图和平台隐藏用例仍属于评分环境待执行门禁。本文件只记录已发生事实，不把受限验证写成已完成。

## 1. 实施方法

采用“预生成的原生 HarmonyOS 成品仓 + 确定性迁移/验证工具 + 有边界 Agent”的双轨方案：

- 客观得分主轨：在 `work/` 直接完成 Stage/ArkTS/ArkUI Jetsnack，逐项对齐功能、界面、接口和规范。
- 泛化与过程轨：以 Skill、迁移台账、稳定组件 ID、Journey 和验证脚本保存可复用迁移方法。
- 裁判阶段：验证已有最佳工程，必要时只做有限、可回退修复，不从零随机生成。

详细设计见 [Jetsnack 鸿蒙化交付设计](superpowers/specs/2026-07-11-jetsnack-harmonyos-delivery-design.md)，研究依据见 [自动迁移调研与解题路线](安卓APP鸿蒙化自动迁移调研与解题路线.md)。

## 2. 基线与已确认合同

- Android 基线来自 `competition` 分支、提交 `23e1421b72b602d80486777efbf24dd248abf3bb`。最终 `work/` 已移除 Gradle/Android 工程层，只在 `source-facts/android-source/` 留存 264KB 的必要源码/vector 证据。
- Android 基线已能离线构建 APK；这只证明源工程可运行，不代表鸿蒙交付完成。
- 正式平台入口仅使用根目录大写 `INSTRUCTION.md`；提交包不再包含非规范的小写兼容文件。
- 本赛题提交包顶层严格只有 `INSTRUCTION.md + work/`；其他赛道可能要求的 `result/`、`logs/` 不适用于本赛题，不混入 ZIP。
- 平台五次独立运行，客观分同时看最好一次和五次全通过；裁判 Docker 无外网。

## 3. Android 功能事实

已完成源码级盘点：

- 页面：Feed、Search、Cart、Profile、Snack Detail、Filter 弹层。
- 主 Tab 路由：`home/feed`、`home/search`、`home/cart`、`home/profile`。
- 详情路由：`snack/{snackId}?origin={origin}`；Cart 深链为 `https://jetsnack.example.com/home/cart`。
- 数据全部为本地静态数据，无 REST/数据库；共有 28 个商品、5 个 Feed 集合和 3 条初始购物车订单。
- 关键视觉：Montserrat/Karla 字体、36 张食品 JPEG、品牌紫 `#4B30ED`、辅色 `#86F7FA`、圆形图片、自定义底栏和渐变。
- Android 没有 Compose `testTag`；鸿蒙实现必须补充稳定 `.id()` 和无障碍文本。
- Manifest 注册的 `RecentOrdersWidgetReceiver` 不是预览代码：它展示五组最近订单，按宽度切换无图单列、有图单列和双列，并由标题/行尾动作打开 Cart；该功能已迁移为 Form Kit。

## 4. 环境盘点

截至 2026-07-11 当前实测：

- 本机 macOS arm64；JDK 21.0.11 可用。系统 Node 为 v25.9.0，最终脚本不把该版本假定为裁判环境合同。
- 已从华为云 HarmonyOS 编译工具镜像取得 Command Line Tools 6.0.0.868。下载包 SHA-256 已与镜像提供的校验文件比对一致：`e07248a8e314f2d9ff3b606ab29748eb936bd69691b1b3715569802864ceef7a`。但包内 `version.txt` 明确标注 `linux-x64`，它不是官方 macOS ARM64 完整工具包。
- 其中平台无关的 Node/Hvigor 部分在本机已实测：`hvigorw --version` 为 6.20.0，`ohpm --version` 为 5.3.2；这不推导出同包内原生工具可在 macOS ARM64 使用。
- 已取得并解包 OpenHarmony 6.0 API 20 Apple Silicon 公共 SDK 6.0.0.48，组件包括 ArkTS、JS、Native、Previewer 和 Toolchains；本地包 SHA-256 为 `c1e61b9b8950a2cb5be79f80abe560322526ef05ecb1d02ce4fd1d9766f194e0`，镜像校验文件的联网比对仍待完成。
- 公共 SDK 内 `hdc -v` 实测为 3.2.0b。Command Line Tools 的版本清单声明 codelinter 6.0.140，但镜像包只带 `bin/codelinter` 启动包装器，没有实际 `codelinter/` 目录，必须从华为登录下载中心取得 Mac ARM 包或安装 DevEco Studio 后才可声称官方静态检查通过；`linter-cli-0.1.0` 只是 x64 Standalone ArkTS checker，不能替代。
- 已把 Hvigor 用户缓存重定向到工作区外的本机工具目录，未修改全局 shell 配置。`tools/preflight.sh` 使用独立临时工作目录探测工具，拒绝把退出码为 0 的错误横幅误判成版本，并把真实执行路径与脱敏诊断路径分开。当前 build-only 实测识别 Hvigor 6.20.0、API 20 SDK 与 HDC 3.2.0b；strict 仍如实因 Code Linter 后端缺失失败。
- 正式工程始终保留 `runtimeOS: HarmonyOS`、目标 API 20、兼容 API 12、phone/tablet。公共 SDK 验证只复制到 `/tmp`，在临时副本中改为 `runtimeOS: OpenHarmony`、API 20 和 `deviceTypes: default`，不会污染正式配置。
- 完整六页面版本已通过公共 API 临时副本的 `CompileArkTS`、资源编译与 HAP 打包。HAP ZIP 容器带构建元数据，包级哈希可能变化，因此不把 SHA-256 当作评分输入或源码确定性依据。这证明所用公共 API 和 ArkTS 语法可编译，但不等价于 HarmonyOS 专有 SDK、签名、官方 Code Linter 或真机验证。
- 官方 HarmonyOS SDK 下载仍涉及华为开发者下载入口；最终工程保持 `runtimeOS: HarmonyOS`。公共 SDK 只用于临时副本的公共 API 交叉编译，结果不得冒充 HarmonyOS 最终构建。
- 已下载官方 HarmonyOS API 20 Codelab 到临时目录，仅用于核对工程骨架和工具链；不会把其业务源码复制进交付件。
- OpenHarmony API20 SDK 内含 Apple Silicon 原生 Previewer，交叉构建 HAP 也包含 `ets/modules.abc`；但该引擎依赖 DevEco 通过 WebSocket、pages、资源路径和设备参数编排，公开资料没有可审计的独立无头截图命令。本机缺少 DevEco 侧启动配置，因此不能把二进制存在或直接启动尝试写成截图验证通过。

官方工具入口：[HarmonyOS Command Line Tools](https://developer.huawei.com/consumer/cn/download/command-line-tools-for-hmos)、[命令行工具获取指导](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/ide-commandline-get)、[Code Linter 命令说明](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/ide-command-line-codelinter)。

## 6. 关键决策记录

1. 使用原生 Stage + ArkTS + ArkUI，不采用 Web/Flutter 包装。
2. 目标工程优先兼容 API 12、以 API 20 编译/目标配置开始；裁判 SDK 合同公开后再锁定最终值。
3. 商品内部 ID 固定化，保持名称、顺序、价格和可见行为，不复制 Android 随机 ID。
4. 原 Android 明确 no-op 的按钮保持 no-op，避免擅自改变隐藏用例状态。
5. 本地图片同步加载，动态渐变使用确定性首帧或有限动画，减少五轮截图漂移。
6. `INSTRUCTION.md` 是平台主 Agent Prompt；`tools/verify.sh` 是唯一非交互执行入口。
7. 自定义字体使用 API 20 推荐的 `UIContext.getFont().registerFont`，每个字重注册独立 family，并用模块级 guard 避免重复注册；参考[官方 HarmonyOS 字体样例](https://gitee.com/harmonyos_samples/text-display-font/blob/master/entry/src/main/ets/common/CommonUtils.ets)。
8. 颜色通过 `base/element/color.json` 与 `dark/element/color.json` 同名资源覆盖，ArkTS 不散落十六进制颜色；集合用枚举，商品查找显式返回 `undefined`，Cart 冷启动通过工厂复制，避免隐藏共享状态。
9. Feed 的七个关键图标保留 Android vector 原始 viewport/pathData，SVG 只转换容器和 tintable fill；高亮卡首帧使用跨六个 `186vp` 槽位的五段对称渐变，确保初始截图不再逐卡重复。
10. 整屏不使用 `accessibilityGroup(true)`，避免吞掉 `nav.*`、Filter 和商品卡节点；原 Android no-op 图标保留 48vp 点击接口但不改变可见业务状态。
11. Cart 恢复源码中的固定顶部栏、侧滑删除反馈、同行价格/数量器和右半宽矩形 Checkout；Filter 恢复 450vp 面板、整面板滚动、三种排序图标与 4+4+5 原始选项；Detail Hero 使用 300→150vp 响应式图片、351→56vp 标题吸顶和对角模糊渐变。
12. Cart 的 `https://jetsnack.example.com/home/cart` 已声明为 browsable skill，Ability 冷启动和 `onNewWant` 都把入口路由同步到 ArkUI 根状态。
13. `INSTRUCTION.md` 新增幂等 Agent 执行协议；10 个动作映射均在对应目标页面暴露真实稳定 ID，避免接口用例只能读到清单却找不到自动化节点。
14. 新增 16 条隐藏用例模拟 Journey，覆盖搜索状态边界、Cart 第五次请求失败、Detail 折叠/大字体/窄屏、Filter 重开生命周期（第59项进一步按全局Filter对象修正）、暗色 Feed、RTL、重复深链与系统返回；系统栏前景色随运行时明暗配置更新。详见 `docs/隐藏用例对抗测试与二次优化.md`。
