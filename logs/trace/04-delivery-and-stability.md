# 交付协议与稳定性复盘

## 1. 平台结构复核

最终归档使用同名目录作为唯一 ZIP 根目录：

```text
02_01_纯人机，已开智/
├── INSTRUCTION.md
├── work/
├── result/
│   └── output.md
└── logs/
    ├── interaction.md
    └── trace/
```

`work/` 内没有再包 Android 仓库目录；提交包不依赖外部 `docs/`、用户本地 Skill、Git 远端或临时 SDK。`INSTRUCTION.md` 只负责告诉平台如何定位已完成仓和如何判断就绪；过程证据放在本目录供评委参考。

## 2. 平台结果的事实复盘

### 旧结果一：`docs/log_7_16.txt`

交付审查通过，但复现阶段失败，237 个用例中准确性 230、稳定性 103。日志没有提供具体失败用例名称，因此不能据此认定某个页面或功能必然缺失。复核旧执行说明后，发现它把 HarmonyOS 构建和工具链准备放在 Agent 的主路径，而评分基础环境不保证这些工具；同时说明包含较多开发上下文，容易使不同执行选择不同工作流。

### 旧结果二：`docs/log_7_17.txt`

交付审查和复现阶段均成功，准确性 262/262，但稳定性为 0。这个组合说明工程至少有一次完整评测路径能够被正确读取，风险更集中在独立执行之间的输出/路径/工作流差异，而不是归档结构缺失。

## 3. 稳定性专项修复

### 3.1 只读就绪门禁

`work/tools/platform_ready.sh`：

- 用脚本自身位置定位 `work/`，不依赖当前临时绝对路径；
- 检查 Stage 配置、Entry 页面、迁移报告、映射清单、事实快照、Journey 和 Skill；
- 不生成文件、不编译、不安装、不联网、不要求人工输入；
- 成功输出固定为：

```text
artifact_status=ready
artifact_type=harmonyos_stage_repository
source_commit=23e1421b72b602d80486777efbf24dd248abf3bb
artifact_path=work
```

### 3.2 防止评分时重生成

迁移 Skill 的“完成仓交接”段落和 Agent 元数据都明确：如果 `entry/`、`migration-manifest.json` 和 `migration-report.md` 已存在，就把当前仓作为完成交付进行审计，不从 Android 重新生成，不改写源码。这样保留了 Agent/Skill 的泛化交付价值，同时避免主评分路径把稳定成品重新变成随机生成任务。

### 3.3 固定相对产物位置

平台复现只交接 `artifact_path=work`，不把临时解压目录、任务目录或绝对机器路径写入产物位置。就绪输出在独立解压的只读副本中逐字节一致，消除了一个可观察的跨执行差异来源。

### 3.4 把慢速构建从主交接路径移开

工程编译方式仍写在 `INSTRUCTION.md`，并保留 `VERIFY_BUILD_TIMEOUT_SECONDS` 调整入口；但平台复现的完成判定是只读就绪脚本，而不是要求基础镜像临时安装 HarmonyOS SDK/Hvigor。这样不会因为平台没有专有工具链而把已经完成的源码仓判为执行失败。

## 4. 最终归档审计

最后一次打包后执行了：

- ZIP UTF-8 文件名和同名根目录检查；
- 必选路径检查；
- `.git`、构建/依赖/Python 缓存、HAP、pyc、`.DS_Store` 和符号链接检查；
- `unzip -t` 完整性检查；
- 包内 `INSTRUCTION.md` 无内部临时执行目录和多轮评分术语；
- 多份独立解压副本以只读权限执行 `platform_ready.sh`，输出完全相同；
- 源工作区与归档内 `work/` dry-run 同步无差异。

最终归档：`02_01_纯人机，已开智.zip`，328 个条目；文件指纹由最终打包命令输出，指纹不是平台评分参数。

## 5. 对评委的诚实边界

上述复核证明的是：提交件结构正确、入口确定、源码输入稳定、过程证据完整、可在可用工具链中执行本地门禁。它不等同于官方隐藏用例满分，因为本地没有平台的专有 Code Linter、真实设备和隐藏测试数据。

稳定性修复针对已观察到的执行交接差异，不能在本地替平台预言最终交集；但它将平台主路径从“每次重新决策/重新生成”收敛为“同一完成仓的只读评分”，是目前证据最充分且风险最小的改进。
