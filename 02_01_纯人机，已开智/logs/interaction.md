参赛者提供了赛题要求、固定 Android 代码仓节点、平台交付规范及其他赛题的评分记录，并持续明确以下总体目标：

- 以 `23e1421b72b602d80486777efbf24dd248abf3bb` 为唯一 Android 基线，完整迁移 Jetsnack；
- 同时覆盖功能一致性、界面截图相似性、接口一致性和鸿蒙代码规范，不以“能够编译”作为唯一完成标准；

- 将官方 Code Linter、编译和静态规则作为规范门禁；本地缺少官方后端时如实记录，不用替代脚本伪造通过结果。

## 2. 参赛者确认或纠正的实施方法

实现过程中，参赛者对方案进行了多轮确认和修正：

- 最终鸿蒙代码仓直接位于 `work/`，不再嵌套 `demo-Jetsnack-android/`；
- 作品通过 ZIP 提交评分平台，不执行 Git push；
- 采用“预生成完整原生鸿蒙仓 + 迁移 Skill + 事实清单 + Journey 门禁”的双轨方案；
- 批准动态系统安全区、响应式布局及 Recent Orders Form Kit 的设计和实现方向；
- 要求以五个独立 Executor 各自运行完整用例集的方式进行本地复现，不把五个裁判误解成五种评分维度；
- 要求 `INSTRUCTION.md` 面向 OpenCode/GLM 裁判保持简短，只描述输入、执行工作流、完成判定和产物；
- 要求慢速评测环境下延长构建等待，Hvigor 默认硬超时因此提高到 1800 秒；
- 根据平台实际结构报错，补齐 `result/output.md`、`logs/interaction.md`、`logs/trace/`，并使用同名 UTF-8 作品根目录打包。

## 2.1 针对 2026-07-16 平台结果的修正决策

平台日志 `docs/log_7_16.txt` 显示交付审查通过，但三个 Executor 均为 `reproduction_status=failed`；237 个用例中准确性通过230、稳定性通过103。日志未返回失败用例名称，因此不把它误判为某个页面或按钮一定缺失。复现旧交付说明后确认：它把构建门禁写成裁判的必做步骤，而平台基础镜像不保证 HarmonyOS SDK；同时 Python 3.9 会在 `zip(..., strict=True)` 处直接失败。

据此确定以下优化方向并执行：

1. 参赛者向组委会再次核实后明确：`INSTRUCTION说明与样例.md` 的具体内容不作为本题执行依据，只参考输入、步骤、产物和完成判定的表达格式；最终说明必须以本题赛题要求和平台交付规范为准。
2. 根 `INSTRUCTION.md` 先执行仓内静态合同与规范门禁；存在 HarmonyOS API 20 SDK 和 Hvigor 时执行真实构建，并以 `BUILD SUCCESSFUL`、`status=passed` 和非空 HAP 共同判定成功。平台镜像缺少 SDK 时不伪造通过、不临时联网安装，继续把已交付的 `work/` 交给 HarmonyOS 评分 Skill。
3. 新增 `work/migration-report.md`，以源码路径索引页面、路由、AppStore 状态、稳定 ID、资源/字体、Form Kit、深链、系统避让区、区域货币和 ArkTS 规范，降低 GLM/OpenCode 在五次独立执行中的探索分歧。
4. 本地构建和合同测试保留为交付前自检；平台 Agent 不从零重写已交付工程，也不递归启动五裁判模拟器。
5. 修复合同测试的 Python 3.9 兼容性，并让五 Executor 归档预演器强制检查迁移报告；269 项本地合同、静态门禁和 API 20 构建均通过。

该修正只改变裁判可发现性、环境兼容性和交付协议，没有针对未知用例添加硬编码答案；真实隐藏用例成绩仍以平台复评为准。
